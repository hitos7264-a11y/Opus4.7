Adaptive Reasoning & xhigh

effort は 5 段階に拡張され、budget_tokens は廃止。 Claude Code の既定は全プランで xhigh に昇格。temperature 系は固定化。

effort の 5 段階

トークン消費を最小化したいとき。FAQ 型チャット、定型分類、単発要約など。
多くの SaaS 統合・ビジネスフォーム処理での妥当な出発点。 レイテンシとコストのバランスが取りやすい。
従来の「しっかり考える」層。長文解析、仕様書からの逆算、 中規模のコード修正など。
high と max の中間。grep、複数ファイル読取、依存追跡などの 探索的ツール呼び出しがより積極的に起動する。 30 分以上走るエージェント運用で high より明らかに道を見失いにくい、 と APIYI の実運用ガイドも記している。Claude Code では全プランで既定。
ボトムライン評価・論文推論・最上位レビュー向け。 ただしコスト・レイテンシの負担が最大。

Breaking Changes

REMOVED
thinking.budget_tokens が廃止
思考の配分は adaptive thinking に一本化。 effort はその総量を決める唯一のダイヤルとなり、xhigh は 「常に深く考える、拡張的な探索を伴う」挙動に固定。
LOCKED
temperature / top_p / top_k 固定化
既定以外の値を投げると 400 エラー。 サンプリングはモデル側で最適化。
従来コードで明示指定していた箇所は事前に削除しておくこと。
HIDDEN
思考テキストは既定で非表示
display: "summarized" 相当のオプトインをしない限り chain-of-thought テキストは表示されない。 ログ基盤・プレイグラウンド UI・デバッグツールの前提を揃える必要がある。
Claude Opus 4.7 は今や adaptive reasoning 一択になった。 — Ethan Mollick（LinkedIn）

コードサンプル: 4.6 → 4.7

BEFORE (Opus 4.6) 
client.messages.create({
  model: "claude-opus-4-6",
  max_tokens: 4096,
  temperature: 0.7,      // ← 4.7 ではエラー
  top_p: 0.9,            // ← 4.7 ではエラー
  thinking: {
    type: "enabled",
    budget_tokens: 8000, // ← 4.7 では廃止
    display: "full"
  },
  messages: [...]
});
AFTER (Opus 4.7) 
client.messages.create({
  model: "claude-opus-4-7",
  max_tokens: 6144,      // 新トークナイザ分の余裕
  // temperature / top_p / top_k は触らない
  thinking: {
    type: "enabled",
    effort: "xhigh",     // low / medium / high / xhigh / max
    display: "summarized"
  },
  messages: [...]
});

自己検証する推論

Anthropic は 4.7 を「長時間タスクを厳格さと一貫性を持って扱い、指示に精密に従い、 報告する前に自分の成果を検証する方法を自ら編み出すモデル」と説明する。

この自己検証はベンチマーク上のハルシネーション率を下げるだけでなく、 長時間エージェントが途中で「自分が何をやっていたか」を見失わないための 内部スキャフォールディングとしても機能する。結果として、 同じ問題をより少ない出力トークンで解く傾向も観察されている。
